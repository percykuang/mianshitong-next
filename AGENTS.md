# AGENTS.md（mianshitong-next）

目的：约定本仓库的人机协作方式、前端实现边界与代码风格，减少重复沟通，避免每次迭代都重新建立同一套上下文。

## 基本约定

- 总是使用中文沟通；代码注释按需，优先简洁。
- 当涉及第三方库、框架、组件库或其 API 的代码编写时，优先使用 Context7 MCP 查询最新文档，再落代码，避免沿用过时写法。
- 对当前任务能直接完成的工作，优先直接实现，不只停留在分析或建议层。
- 做改动时优先局部、精准、低风险，不为“顺手优化”引入与任务无关的大范围重构。

## Monorepo 协作约定

- `apps/web`、`apps/admin` 放应用入口与页面编排；可复用能力优先沉淀到 `packages/*`，不要在 app 内无序复制。
- 新增共享组件前，先判断是否应放入 `packages/ui`；新增图标前，先判断是否应放入 `packages/icons`；新增设计变量前，先判断是否应放入 `packages/tokens`。
- 对同一语义的资源、配置或常量，优先保持单一数据源，避免在 `app`、`public`、`packages` 中出现多份内容长期并存。
- 修改共享包时，默认兼顾 `web` 与 `admin` 两端的使用场景，不要只为单个页面临时定制公共 API。

## Next.js 与资源约定

- 优先遵守 Next.js App Router 的文件约定与 Metadata 机制，不绕开框架默认能力实现页面、图标、元信息和静态资源。
- 普通页面展示资源优先放在 `public/` 下，通过稳定静态路径访问；只在确实需要 App Router metadata file convention 时，使用 `app/icon.*`、`app/opengraph-image.*` 等特殊文件。
- 对本地图片资源，优先使用 `next/image`；对 SVG 等不需要优化的资源，可使用 `unoptimized`，但仍优先保持 `Image` 接口一致性。
- 外链默认补齐 `target="_blank"` 与合适的 `rel` 属性，避免安全与行为问题。

## Tailwind CSS 约定

- 始终优先使用 Tailwind CSS v4 的 canonical 写法，避免继续引入已废弃或非推荐语法。
- Important modifier 一律使用后缀 `!`：
  - `h-11!`
  - `hover:border-blue-500!`
- CSS 变量一律使用 v4 简写语法，而不是旧的 `var(...)` 包裹写法：
  - `text-(--token)`
  - `bg-(--token)`
  - `border-(--token)`
  - `shadow-(--token)`
- RGB arbitrary value 优先使用 slash 语法：
  - `rgb(22_119_255/0.14)`
- 若某个值可以无损映射到 Tailwind 官方 scale，优先使用 scale class，而不是 arbitrary value：
  - `rounded-4xl` 优先于 `rounded-[32px]`
- 若替换为 scale 会改变视觉效果，则不要机械替换 arbitrary value；需要先确认设计意图，例如 `rounded-[28px]` 这类介于 scale 之间的值。

## UI 与样式最佳实践

- 优先复用 `packages/ui`、`packages/tokens`、`packages/icons` 中已有能力，再新增页面级实现。
- 页面应尽量围绕语义 token 设计，不要在业务代码里扩散一批难以维护的硬编码颜色、阴影、圆角值。
- 样式变更优先保持同一页面内部的视觉语言一致：圆角、边框、阴影、留白、交互反馈应成体系，而不是局部“好看”。
- 对明显重复的 UI 片段，优先抽成小而稳定的组件，而不是在多个页面复制 className。
- 能用现有布局与组件组合解决的问题，不新增只为单处服务的“伪公共组件”。

## 代码实现最佳实践

- React 组件与自定义 Hook 优先使用普通函数声明，保持模块主结构清晰，并与 React 官方示例风格一致。
- 事件处理函数、回调函数、短小局部逻辑优先使用箭头函数；仅在局部使用的短工具函数也可使用箭头函数。
- JSX 中的事件处理器，若不依赖当前循环项、局部闭包参数或事件对象的额外编排，优先提取为组件顶部的命名函数，避免在 JSX 中堆叠一层层匿名函数。
- 对列表渲染中需要绑定当前项参数的短处理器，或必须同时消费 event 与当前局部上下文的简单逻辑，可保留 inline 箭头函数；不要为了“消灭 inline”机械引入函数工厂或让代码更绕。
- 若某个 `render`、`menu`、`columns` 或 JSX 分支内已经出现成段匿名回调逻辑，优先提取为命名处理器；当一整块交互逻辑持续膨胀时，优先拆成小组件而不是继续堆在当前 JSX 里。
- 模块级导出的公共函数若承担明确语义职责，优先使用普通函数声明，避免项目中长期混用“看心情”的函数风格。
- 不要在组件内部定义新的组件或自定义 Hook；组件和 Hook 应定义在模块顶层，避免状态重置和可读性下降。
- React state 应保持最小且避免冗余；能通过 props、现有 state 或常量直接计算出的值，不要再额外存一份 state。
- 不要用 Effect 维护派生数据或串联 state 变更；能在 render 阶段计算的值就在 render 中计算，能在事件处理函数中完成的更新就不要拆到 Effect。
- `useEffect` 优先只用于与 React 外部系统同步，例如请求、订阅、定时器、DOM、副作用型浏览器 API；不要把它当作通用业务流程编排工具。
- 状态应优先就近放置，只在确有共享需求时再提升；不要过早把局部交互状态提升到过高层级或全局 store。
- 新增组件 API 时，优先保持最小而稳定，避免过早暴露过多变体、尺寸和布尔开关。
- 组件命名尽量表达语义与职责，而不是只描述外观。
- 非必要不要把简单逻辑过度抽象成 hooks、utils 或配置对象；但重复三次以上的模式应考虑收敛。
- 优先清晰可读，再考虑“聪明”的写法；避免引入阅读成本高、收益很低的技巧性代码。
- 修改现有实现时，先顺应当前仓库风格，再做少量提升，不用每次都带入另一套个人偏好。

## 编码四大原则

请严格遵守以下四条编码原则生成代码：

1. 先思考，再编码

不主观臆测，不掩盖困惑，明确权衡利弊。

大模型往往会默默选择一种理解并直接执行。本原则要求进行显式推理：

- 明确说明假设 —— 不确定时，主动询问而非猜测
- 提供多种解读 —— 存在歧义时，不私自做决定
- 合理提出异议 —— 若有更简单的方案，直接指出
- 困惑时暂停 —— 指明不清楚的地方并请求澄清

2. 简洁优先
   用最少的代码解决问题，不做任何 speculative（预判性 / 多余）设计。

抵制过度工程化的倾向：

- 不实现需求之外的功能
- 单次使用的代码不做抽象封装
- 不添加未被要求的 “灵活性” 或 “可配置性”
- 不为不可能出现的场景做异常处理
- 如果 200 行能精简到 50 行，就重写
- 检验标准：资深工程师会不会觉得这过于复杂？如果会，就简化。

3. 精准修改

只修改必须改动的部分，只清理自己造成的问题。

修改现有代码时：

- 不 “优化” 相邻代码、注释或格式
- 不重构没有问题的代码
- 遵循现有风格，即使你有不同写法
- 发现无关的死代码，只需指出，不要删除

当你的修改产生无用代码时：

- 移除因你的改动而不再使用的导入、变量、函数
- 除非被要求，否则不删除原本就存在的死代码
- 检验标准：每一行被修改的代码，都必须直接对应需求。

4. 目标导向执行

先定义成功标准，循环验证直到达成。

把指令式任务转化为可验证的目标：

- 把 “添加校验” → 改为 “为非法输入编写测试并让测试通过”
- 把 “修复 Bug” → 改为 “写出能复现 Bug 的测试，再让测试通过”
- 把 “重构 X” → 改为 “确保重构前后测试都能正常通过”

多步骤任务，先给出简要计划：

1. [步骤] → 验证：[检查项]
2. [步骤] → 验证：[检查项]
3. [步骤] → 验证：[检查项]

## 质量检查

- 对实际代码改动，交付前至少运行与任务相关的检查。
- 优先使用这些命令：
  - `pnpm format:check`
  - `pnpm lint`
  - `pnpm typecheck`
- 若改动影响页面视觉、资源加载、导航或交互，默认至少做一轮对应页面的本地验证。

- 若后续把某条协作约定升级为工具链强约束，应同步更新本文件，说明“软约束”已升级为“硬约束”。
